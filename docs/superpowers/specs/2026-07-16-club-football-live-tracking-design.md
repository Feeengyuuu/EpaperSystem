# SportsDashboard 五大联赛实时追踪设计

**日期**：2026-07-16
**状态**：已实现；2026-07-19 按新赛季要求将默认入口改为五大联赛
**范围**：实现、验证并部署到 `ColoredEpaperFrame`

## 1. 目标

将 SportsDashboard 左上角默认入口直接切换为欧洲五大俱乐部联赛页面，支持英超、西甲、德甲、意甲和法甲，并达到当前世界杯页面相同级别的实时追踪能力。

首期必须同时满足：

- 保留世界杯模块和手动/自动模式，但新赛季默认不再自动进入世界杯。
- 支持五个联赛，而不是只做英超或西甲试点。
- 实时比赛优先，其次是最近即将开赛的比赛，再次是最新赛果。
- 同等级比赛在联赛之间轮换，避免单一联赛长期占据焦点主卡。
- 使用真实、透明、比例正确的联赛 Logo 与球队队徽，并在本机缓存。
- 网络或单个数据源失败时不显示空白页，也不把推断状态冒充实时比分。
- 首期只在本机完成代码、UI、数据链接和刷新逻辑验证；设备部署另行确认。

## 2. 已确认的设计选择

### 2.1 联赛范围

| 展示名称 | football-data.org | ESPN |
| --- | --- | --- |
| 英超 | `PL` | `eng.1` |
| 西甲 | `PD` | `esp.1` |
| 德甲 | `BL1` | `ger.1` |
| 意甲 | `SA` | `ita.1` |
| 法甲 | `FL1` | `fra.1` |

默认五个联赛全部启用。设置页允许以后关闭个别联赛，但首期不支持添加五大联赛之外的任意联赛代码。

### 2.2 页面选择

默认采用 `club`，同时保留可选的自动模式和手动覆盖：

- `auto`：按世界杯有效窗口自动选择世界杯或俱乐部联赛。
- `worldcup`：始终选择世界杯模块。
- `club`：始终选择五大联赛模块，也是缺省值和非法配置的安全回退。

自动模式下，世界杯从首场比赛前 14 天开始生效，在决赛结束后继续保留 24 小时，之后自动回到五大联赛。未来世界杯进入相同窗口时自动再次切回世界杯。

自动判断优先使用新鲜赛程，其次使用最后有效世界杯赛程缓存。没有可信世界杯赛程时选择俱乐部联赛，不能因为一次请求失败而永久卡在世界杯模式。

### 2.3 UI 方案

采用已批准的 B 方案：左侧焦点主卡加右侧五联赛纵轨。

- 页面仍是完整的 `800×480` SportsDashboard。
- 只替换世界杯实际槽位；参考尺寸为 `536×240`，当前生产实例的实际槽位为 `552×208`。
- 右侧 LPL 区域和下方 NBA 区域不改变位置、尺寸或绘制逻辑。
- 俱乐部面板必须按运行时实际槽位原生排版，禁止先绘制 `536×240` 再做非等比整体缩放。
- 左侧显示当前焦点比赛的大队徽、比分、状态、场地、更新时间和来源状态。
- 右侧固定五行，每个联赛一行，显示联赛 Logo、双方小队徽、队名和比分或开赛时间。
- 右侧纵轨固定按英超、西甲、德甲、意甲、法甲排序；焦点所在联赛使用高亮状态。
- 列表中的左队队名靠左，右队队名靠右，围绕中间比分或 `VS` 完全镜像。
- 队徽和联赛 Logo 使用固定边界框内的 `contain` 规则，不拉伸、不裁切、不改变宽高比。
- 正文、队名和元数据使用适合彩色墨水屏的深色粗体或半粗体，避免细字和浅灰字。
- 首期不在足球面板中加入完整积分榜或额外排名栏；积分榜只在数据层缓存，不能挤压已批准的比赛与 Logo 布局。

批准的本机 PNG 基准为 `output/playwright/club-football-b-preview.png`。该图是视觉方向参考；实现必须通过坐标和溢出测试，而不是通过截图缩放来逼近。

## 3. 架构

### 3.1 模块边界

新增两个独立模块：

- `sports_dashboard/club_football.py`：设置解析、数据获取、缓存、规范化、合并、选择、自动窗口和实时状态。
- `sports_dashboard/club_football_render.py`：负责按照实际槽位计算焦点卡、五行联赛轨道与 Logo 布局。

新增 `ClubFootballMixin` 与 `ClubFootballRenderMixin`，由现有 `SportsDashboard` 组合。不得把五大联赛逻辑继续塞入 `worldcup.py` 或 `worldcup_render.py`。

现有世界杯代码和缓存文件保持兼容。页面选择在组合渲染入口完成：先根据 `footballPanelMode` 选择世界杯或俱乐部模块，再把选中的左上面板与现有 NBA、LPL 等区域组合。

### 3.2 双数据源职责

选择已批准的双数据源方案：

- football-data.org：赛程、比赛日期、完成状态和积分榜。
- ESPN：实时比分、比赛分钟、实时状态、联赛 Logo、球队队徽和补充赛程。

football-data.org 使用 `/v4/competitions/{code}/matches` 与 `/v4/competitions/{code}/standings`；ESPN 使用 `/apis/site/v2/sports/soccer/{slug}/scoreboard`。所有具体 URL 由固定基础地址和上表白名单代码构造，设置值不能直接成为任意请求 URL。

ESPN-only 方案虽然简单，但没有可靠后备；API-Sports 方案存在免费套餐和赛季权限限制，均不作为首期主架构。

### 3.3 统一事件模型

两个来源进入渲染前统一为内部事件对象，至少包含：

- `league_code`、`league_name`、`espn_slug`
- `event_key`、`start_utc`、`status`
- `home_name`、`away_name`、规范化名称和别名集合
- `home_score`、`away_score`、`display_clock`
- `venue`、`matchday`
- `league_logo_url`、`home_logo_url`、`away_logo_url`
- `provider`、`source_state`、`fetched_at`
- `provider_status_confirmed`、`inferred_live_window`

事件合并先限定在同一联赛，再使用开赛时间容差和主客队别名匹配。默认开赛时间容差为 15 分钟。主客队不能因名称别名而交换；如果只能反向匹配，必须显式翻转比分和队伍字段并由测试覆盖。

内部 `event_key` 由联赛、UTC 开赛时间和规范化主客队构成，不能直接依赖某一家提供商的事件 ID。

## 4. 选择与轮换

每个启用联赛先选出一条用于右侧纵轨的最佳事件：

1. 提供商确认的直播比赛。
2. 最近即将开始的比赛。
3. 最新完成的比赛。
4. 没有比赛时显示带联赛 Logo 的“暂无赛程”行。

焦点主卡在五个联赛候选中采用相同优先级。多个候选处于同一优先级时，使用持久化轮换状态避免立即重复，并保证五个联赛最终都能获得焦点机会。

开赛时间只能用于启动轮询窗口，不能单独把页面标记为 `LIVE`。只有提供商确认直播时才显示实时比分标签；否则使用“即将开始”或“状态确认中”。

## 5. 自动切换

新增纯函数式选择逻辑，输入当前时间、世界杯赛程摘要和手动模式，输出 `worldcup` 或 `club`。

`auto` 模式规则：

1. 找到可信的世界杯首场与决赛时间。
2. 当前时间在“首场前 14 天”到“决赛完成后 24 小时”之间时选择世界杯。
3. 决赛尚未确认完成时，可用最后一场计划结束时间加安全窗口，但页面必须继续显示数据来源状态。
4. 世界杯赛程源暂时失败时使用最后有效缓存。
5. 没有可信新鲜或缓存赛程时选择俱乐部联赛。

手动模式直接覆盖以上规则，但不修改或删除另一模式的缓存。

## 6. 刷新与缓存

### 6.1 普通状态

- football-data.org 赛程默认缓存 6 小时。
- ESPN 非比赛日赛程使用长缓存，不进行分钟级轮询。
- 积分榜在普通日期使用长缓存。
- Logo 下载成功后写入本机缓存，渲染线程只读取本地文件。

### 6.2 比赛日状态

- 比赛日前后由 ESPN 每 15 分钟检查一次状态。
- 积分榜在比赛日最多每小时刷新一次。
- 只更新已启用联赛，关闭的联赛不得产生请求。

### 6.3 直播状态

- 从开赛前 15 分钟开始进入快速状态检查窗口。
- 提供商确认直播后，相关联赛的数据和缓存图像按 60 秒间隔刷新。
- 其他没有直播的联赛继续使用普通缓存，避免每分钟请求全部五个联赛。
- 连续比赛或紧邻开赛必须延长 `live_until`，不能在下一场开始边界停止刷新。
- 比赛完成并且没有相邻比赛后退出快速刷新。

新增 `club_football_live_state.json`，版本独立于 `worldcup_live_state.json`。`SportsDashboard.get_live_refresh_state()` 将其作为新的活动来源，复用现有 `refresh_task` 快速体育缓存刷新路径。

实时状态文件至少包含版本、`has_live`、`live_until`、活动联赛、选中事件摘要、来源、新鲜度和生成时间。写入必须采用现有原子 JSON 缓存模式。

## 7. Logo 与队徽

首选 ESPN 当前联赛与球队记录返回的透明 PNG。首次下载时验证：

- HTTP 状态成功。
- 内容类型是受支持的图片。
- 实际图片可以解码。
- 宽高和文件大小在安全限制内。

缓存键由来源 URL 与规范化球队或联赛身份组成。URL 变化时允许更新，但一次网络失败不能删除现有最后有效 Logo。

渲染优先级：

1. 已缓存的 ESPN Logo。
2. 已验证的本地覆盖资源。
3. 现有通用球队 Logo 回退。
4. 高对比度文字缩写占位符。

所有 Logo 均在固定矩形中等比缩放。主卡队徽与纵轨小队徽分别使用独立尺寸预算；不得把主卡队徽先缩成小图再二次放大。

## 8. 故障降级与数据诚实性

- ESPN 失败但 football-data.org 可用：显示赛程与最后有效比分，标记“缓存数据”，不显示未经确认的实时分钟。
- football-data.org 失败但 ESPN 可用：继续显示 ESPN 赛程与比分，积分榜保留缓存或隐藏，不能伪造积分。
- 两个来源都失败：显示最后有效完整页面、来源状态和更新时间。
- 没有任何有效缓存：显示结构化“暂无可用比赛数据”面板，仍保留五个联赛 Logo 或文字占位，不能返回空白图。
- 单个 Logo 失败：只回退该 Logo，不让整个面板生成失败。
- 缓存 JSON 损坏：忽略损坏文件，重新获取；重新获取也失败时使用可验证的上一代缓存或结构化无数据面板。
- 所有网络请求必须有超时，遵循现有全局代理清理和请求预算规则。

渲染中的 `LIVE` 只表示提供商确认的直播状态。基于开赛时间的本地推断只用于调度刷新，并在数据对象中以 `inferred_live_window` 单独保存。

## 9. 设置

设置页新增最小必要字段：

- `footballPanelMode`：`auto`、`worldcup`、`club`，默认 `club`。
- `clubFootballEnabledLeagues`：以逗号分隔的白名单代码，默认 `PL,PD,BL1,SA,FL1`；设置页将其渲染为五个启用复选框。
- `clubFootballLiveRefreshEnabled`，默认开启。
- `clubFootballLiveRefreshIntervalSeconds`，默认 60，范围沿用现有体育实时刷新限制。

14 天前置窗口、24 小时尾部窗口、15 分钟比赛日检查和 6 小时普通缓存首先作为代码默认值，不在首期设置页暴露，避免设置页继续膨胀。现有 football-data.org 密钥配置继续复用，不新增重复密钥字段。

## 10. 测试与本机验收

### 10.1 单元测试

在现有 SportsDashboard 测试中覆盖：

- 五个 football-data.org 代码与 ESPN slug 的映射。
- 两个来源的解析、规范化、主客队合并和反向顺序处理。
- `LIVE > UPCOMING > RECENT` 选择优先级。
- 同优先级联赛轮换、公平性和防立即重复。
- 世界杯首场前 14 天、决赛后 24 小时的边界。
- 三种手动模式覆盖。
- 直播状态写入、过期和连续比赛窗口延长。
- 来源失败、损坏缓存和无数据降级。
- Logo 下载验证、最后有效缓存和占位符。
- `536×240` 参考尺寸与 `552×208` 生产尺寸均原生绘制，所有坐标和文字不越界。
- 右侧队名靠右、左侧队名靠左，并围绕比分镜像。
- Logo 使用 `contain` 且不发生非等比拉伸。

### 10.2 调度测试

在 RefreshTask 测试中覆盖：

- `club_football_live_state.json` 活动时返回 60 秒快速刷新。
- 非直播或已过期时不触发快速刷新。
- 连续比赛不会在边界停止。
- 资源压力保护仍然生效，同时沿用 SportsDashboard 的高优先级刷新路径。
- 关闭实时刷新设置后不启动快速刷新。

### 10.3 网络与链接烟雾测试

提供显式、本机运行的非 CI 烟雾检查：

- 五个 ESPN scoreboard URL 返回成功 JSON。
- 返回的联赛 slug 与预期一致。
- 样例联赛 Logo 和球队队徽 URL 返回可解码图片。
- football-data.org 在已配置密钥时可读取五个联赛；无密钥时明确跳过而不是误报通过。

固定单元测试使用保存的最小化 fixture，不把公网可用性变成默认 CI 的不稳定依赖。

### 10.4 视觉验收

- 生成完整 `800×480` PNG，而不是单独放大俱乐部面板。
- 与当前 SportsDashboard 基准比较，确认只有左上 `536×240` 区域变化。
- 检查右侧 LPL 和下方 NBA 区域像素未被移动或缩放。
- 检查所有五行可读、右侧队名镜像靠右、Logo 无裁切。
- 在彩色墨水屏调色板模拟或量化后检查细字和低对比度退化。

完成标准：聚焦 SportsDashboard 与 RefreshTask 测试通过，网络烟雾检查通过或以“无密钥”明确跳过，最终完整 PNG 获得确认，且 `git diff --check` 无问题。

## 11. 不在首期范围内

- 五大联赛之外的任意联赛、杯赛或欧冠。
- 博彩赔率、推送通知、比赛事件时间线和球员统计。
- 新建独立 Web 服务或数据库。
- 改动右侧 LPL、下方 NBA 或其他 SportsDashboard 页面结构。
- 部署到 `ColoredEpaperFrame`、修改设备持久化设置、提交发布包或推送远端。

## 12. 实施边界

实施应保持窄范围：新增俱乐部联赛模块、接入现有组合入口和实时刷新契约、扩展设置与测试。不得借机重构整个 SportsDashboard，也不得复制世界杯模块形成第二套无法共享的通用网络与缓存代码。

本机阶段完成并获得最终 PNG 与测试证据后，再单独决定是否打包、部署和进行物理屏幕验证。
